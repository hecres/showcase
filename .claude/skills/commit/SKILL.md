# コミット作成

Conventional Commits に基づく規約に従ってコミットメッセージを作成し、コミットする。

## 参照

- [フォーマット構造](references/git-commit-format.md)
- [description 記述規約](references/git-commit-description.md)
- [type 一覧・選択基準](references/git-commit-types.md)
- [コミット承認フロー](../../rules/engineering/git-operations/commons/git-commit-approval.md)

## 手順

- [ ] 1. 上記の参照ファイルをすべて読み込む
- [ ] 2. **全変更の把握** — `git status` で全変更ファイル（ステージ済み・未ステージ・Untracked）を確認し、`git diff` で未ステージ差分、`git diff --staged` で既ステージ差分の内容をそれぞれ把握する。既にステージされている内容も無視せず、手順4で対象に含めるか取捨選択する
- [ ] 3. **コミット分割判定** — 変更内容が複数の独立したタスクにまたがる場合、コミットを分割する
  - 分割可能な場合: タスクごとにコミット案を `1/N`, `2/N` の形式で提示する
  - 分割困難な場合（同一ファイルを複数タスクで編集している等）: ユーザーにその旨を伝え、まとめてコミットしてよいか確認する
  - 単一タスクの場合: そのまま手順4に進む
- [ ] 4. **コミット対象ファイルをステージング** — `git add <files>` でコミット対象のファイルを明示的に指定してステージングする。既に意図と異なるファイルがステージ済みの場合は `git restore --staged <file>` で外す
- [ ] 5. 変更内容に基づき適切な type を選択する
- [ ] 6. description を記述規約に従って作成する
- [ ] 7. 必要に応じて本文・フッターを追加する
- [ ] 8. セルフレビューを実施する
- [ ] 9. **承認直前の最終確認** — `git diff --staged` を実行し、手順2で把握した「コミット対象とする変更」がすべて含まれ、かつ対象外のファイルが含まれていないことを照合する。漏れ・混入があれば手順4に戻る
- [ ] 10. コミット承認フローに従い、ユーザーの承認を得る（承認スキップが指示されている場合は省略）
- [ ] 11. コミットを実行する（分割コミットの場合はタスクごとに手順4〜11を繰り返す）

## アンチパターン

以下の操作は新規ファイルやステージ意図の取り違えを招くため使用しない。

- `git commit -a` / `git commit -am "..."` — 追跡済み変更のみが自動ステージされ、Untracked（新規ファイル）が漏れる
- `git add .` / `git add -A` — 意図しないファイルまでステージされる。必ず `git add <files>` でファイルを明示する
- 手順4を省略しての `git commit` 実行 — 既ステージ状態の前提が崩れると対象漏れ・混入が起きる

## セルフレビュー

- [ ] `<type>: <description>` の形式になっているか
- [ ] type が type 一覧に含まれているか
- [ ] description が具体的で、第三者が文脈なしで理解できるか
- [ ] セッション固有の一時的な用語が含まれていないか
- [ ] 本文がある場合、what（詳細説明）→ 空行 → why（理由）の構成になっているか
- [ ] 破壊的変更がある場合に `!` または `BREAKING CHANGE:` で明示されているか
- [ ] `git diff --staged` の内容が手順2で把握した変更と一致しているか

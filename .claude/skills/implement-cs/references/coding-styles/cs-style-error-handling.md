# C# エラーハンドリング

エラー条件の分類と防御的コードに関するルールを定める。

---

## 1. エラーの根本原因を隠さない

エラーが発生しうる箇所で、根本原因を隠す防御的コードを書いてはならない。表面的な症状（ゼロ除算、null参照、空コレクション等）を無条件に抑制する処理は、真の問題（データ不備、前提条件の未充足等）の検知を妨げる。

### 1.1 判断基準

「その値は本来ありえるか？」を問う。

| 値の性質 | 対応 |
|---------|------|
| 本来ありえない（データ不備・設定漏れ） | エラーとして報告する。無条件の抑制は禁止 |
| 正常系でありえる（任意入力・外部データ等） | ガード節で正常に処理する |

### 1.2 実行時例外を観測した場合の対応

実行時に `NullReferenceException` 等を観測した際、null チェック追加で症状を抑え込む前に **必ず根本原因を調査する**。

1. **値の出自を遡る**: 例外箇所の値が、本来どのタイミングで non-null になるべきだったかを特定する（コンストラクタ引数・DI 注入・フィールド初期化子・外部 API 戻り値 等）。判定基準は §5 の表を参照
2. **判定結果による分岐**:
   - **本来 non-null** → null チェック追加は症状隠蔽。注入経路・初期化順序・フレームワーク仕様の誤解などを構造的に直す
   - **仕様上 null になり得る** → 準正常系 / 許容異常系 として扱う（§2）

#### DI 注入対象の特殊事情

DI 注入対象（`[Inject]` フィールド・コンストラクタ引数等）が実行時 null になっている場合、注入経路の見直しが最優先。フレームワーク（VContainer 等）の仕様により注入されない条件下にあるなら、注入対象を変える / 取得経路を変える等の構造修正で対処する。

例: VContainer では LifetimeScope 自身の `[Inject]` は `autoInjectGameObjects` 経由でのみ解決される。Configure 内で依存解決したい場合は `RegisterBuildCallback` パターンに切り替える等の構造修正で対処する。

### 1.3 禁止パターン

```csharp
// ❌ 悪い例 — イベントが最低1つ必要な場面でゼロ除算を黙って回避
// eventCount が 0 であること自体がデータ不備だが、それを隠している
if (eventCount == 0) return;
var interval = duration / eventCount;
```

```csharp
// ❌ 悪い例 — 空コレクションを黙って無視
// items が空であること自体が前提条件の未充足だが、それを隠している
if (items.Count == 0) return;
```

```csharp
// ❌ 悪い例 — [Inject] フィールドが null である状態を許容するのは構造的問題の隠蔽
private void Process()
{
    if (injectedService == null) return;
    injectedService.Execute();
}
```

## 2. エラー条件の分類

### 2.1 準正常系

設計上想定されるエラー条件で、正常系とは異なるが処理フローとして定義済みのもの。例外を投げず、専用の通知機構や代替処理で対応する。

```csharp
// ✅ 良い例 — 準正常系: キュー全消費は設計上想定される状態
if (headIndex >= allItems.Count)
{
    // 準正常系: キュー全消費時は通知して終了
    exhaustedCallback?.Invoke();
    return null;
}
```

### 2.2 許容異常系

完全に想定外のバグだが、アプリ実行に致命的な影響を与えないため例外を投げずに継続するもの。**必ずエラーログを出力する**（例外の代替）。ログメッセージの先頭に `[許容異常系]` プレフィックスを付与する。

**適用対象**: リソース内部のデータ不備等。リソース自体は読み込めているが内部データに不備がある場合で、表示やビジュアルが欠落するだけでロジックの整合性は維持される。

**適用外（例外を投げる）**: ロジックの不整合（状態遷移の矛盾、インデックス範囲外、前提条件の未充足等）。ロジックの不整合は根本原因の隠蔽を防ぐため、例外を投げて即座に検知する。

```csharp
// ✅ 良い例 — 許容異常系: リソース不備によるエラー報告 + 処理継続
if (eventCount == 0)
{
    // 許容異常系: イベント未設定は想定外だが演出スキップで継続
    Logger.LogError($"[許容異常系] イベントが未設定です: {resourceName}");
    return;
}

var interval = duration / eventCount;
```

```csharp
// ✅ 良い例 — ロジック不整合は例外で通知（許容異常系にしない）
if (headIndex >= slots.Count)
{
    throw new InvalidOperationException($"HeadIndex ({headIndex}) がSlot数 ({slots.Count}) を超えています。");
}
```

## 3. コメント規約

| 分類 | コメントプレフィックス | エラーログ | 用途 |
|------|---------------------|------------|------|
| 準正常系 | `// 準正常系:` | 任意（通常は不要） | 設計上想定されるエラー条件 |
| 許容異常系 | `// 許容異常系:` | **必須**（`[許容異常系]` プレフィックス付き） | 想定外だが致命的でないバグ |

コメントには「なぜこの分類か」「なぜ例外を投げないか」を簡潔に記述する。

```csharp
// 準正常系: キュー全消費時は通知して終了
// 許容異常系: キュー全消費後の拒否通知は想定外だが影響範囲が狭いため継続
```

## 4. 正常フローの早期 return との区別

正常系として到達しうる条件での早期 return は防御的コードではない。コメントで「フォールバック」「念のため」と記述しないこと。

| 状況 | 分類 | 対応 |
|------|------|------|
| キューが空で操作できない | 正常フロー | 専用の通知機構で呼び出し元に伝達する。コメント不要 |
| 初期化前に呼ばれた | 本来ありえない | 例外を投げる |
| 設計上想定されるエラー条件 | 準正常系 | 代替処理 + `// 準正常系:` コメント |
| 想定外だが致命的でないバグ | 許容異常系 | エラーログ `[許容異常系] ...` + `// 許容異常系:` コメント |

「フォールバック」とコメントされた早期 return を見つけた場合、まず「その条件は本来ありえるか」を問い、正常フローであれば適切な通知機構に置き換える。

## 5. 防御的コードの検証手順

差分内の null チェック・早期 return・デフォルト値返却を検出するため、以下を実行する。**目視で「問題なさそう」と判断せず、必ず一覧化して照合する。**

1. 差分内の null チェック（`== null` / `!= null` / `?.`）・早期 return・デフォルト値返却を列挙する
2. 各チェック対象の値について、**値の出自を遡って** null になりうるかを判定する:

| 出自 | null の可能性 | 対応 |
|------|-------------|------|
| フィールド初期化子 `new()` / `= new Xxx()` | なし（違反） | チェックを除去する |
| `=> field`（`new()` で初期化済みフィールドへの式プロパティ） | なし（違反） | チェックを除去する |
| メソッド引数（非 null 許容） | — | **検証対象外**（`cs-null-checking.md` のガード節で保護する対象。除去しない） |
| 外部 API 戻り値 / ユーザー入力 | あり | チェックは妥当 |

3. null になりえない値へのチェックが見つかった場合、そのチェックを除去する。本来ありえない null を黙って処理すると、初期化漏れ・設定ミス等の真の問題を隠蔽する

### 検証例

```csharp
// ❌ 違反 — QueueProvider は new() で初期化済みの processor への式プロパティ。null にならない
var queueProvider = Model.QueueProvider;
if (queueProvider == null || !queueProvider.IsInitialized) return;

// ✅ 修正後 — null チェックを除去
var queueProvider = Model.QueueProvider;
if (!queueProvider.IsInitialized) return;
```

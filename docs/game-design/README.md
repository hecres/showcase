# 技術設計資料

開発中ゲームの技術設計のうち、公開可能な範囲を抜粋して掲載します。

## 採用アーキテクチャ

開発中ゲームでは以下のアーキテクチャを組み合わせています。<br/>
同一のアーキテクチャでも解釈には幅があるため、本PJでの解釈と、採用する概念の範囲を併記します。

| アーキテクチャ | 本PJでの解釈 | 採用する概念 |
|---|---|---|
| クリーンアーキテクチャ | ドメインを外側の都合から独立させる、依存方向のルール<br/><br/>よく知られた同心円の図はあくまで一例で、層の数や名前そのものは本質ではない。<br/>本質は「依存は常に内側（ドメイン）へ向ける」という方向の制約にある。 | - 依存方向のルール（依存は内側のドメインへ一方向）<br/>- 依存性逆転（インターフェース経由で依存を反転） |
| DDD | ドメイン（対象領域）の知識をモデルで表現し、設計の中心に置く思想<br/><br/>特定のパターン集ではなく、「ドメインを中心に考える」という思想全体として捉える。<br/>本PJでは戦略・戦術の両面から、Unity での実装と相性のよい概念を選んで取り入れている。 | - 戦略的: ユビキタス言語 / 境界づけられたコンテキストの発想<br/>- 戦術的: レイヤードアーキテクチャ / 構成要素（Entities / ValueObjects / Services / Repositories） |
| MVRP | Model と View の分離を目的とする設計パターン<br/><br/>MVP ではイベントを手続き的に配線するのに対し、MVRP では状態ストリームの購読によって Presenter が両者を仲介する。<br/>リアクティブに仲介することで Presenter を薄く保てる。 | - Model / View / Reactive Presenter の三分割<br/>- Model 状態の Observable 公開<br/>- 購読ライフサイクルの管理 |

## 採用アーキテクチャの具体化

採用したアーキテクチャを以下のように具体化しています。

| 具体化方法 | 目的 | ベースとなったアーキテクチャ |
|---|---|---|
| [用語集](../glossaries/README.md) | ユビキタス言語の確立（境界は設けず、本PJ内で一貫した用語定義を行う） | - DDD（ユビキタス言語） |
| [レイヤー構成](architectures/layer-structure.md) | ドメインの独立と依存方向の制御 | - クリーンアーキテクチャ（依存方向のルール、依存性逆転）<br/>- DDD（レイヤードアーキテクチャ、構成要素） |
| [コンテキスト分割](architectures/context-split.md) | コンテキスト境界の明確化とモデルの独立 | - DDD（境界づけられたコンテキストの発想） |
| [ドメインモデリング](architectures/domain-modeling.md) | ドメインモデルの表現と誤用の防止 | - DDD（構成要素） |
| [UI構成](architectures/ui-structure.md) | ドメイン状態と UI のリアクティブな同期 | - MVRP（Model / View / Reactive Presenter の三分割、Model 状態の Observable 公開、購読ライフサイクルの管理） |

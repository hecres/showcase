# 仕様書フォーマット

本ドキュメントは、仕様書（docs/specs/）を読む際に知っておくべき記法・構成のルールをまとめたものである。

---

## 仕様書の基本方針

- 仕様書は**完成形の理想仕様**を記述している。実装状態（未実装・開発中・実装済み等）の情報は含まない
- 記述はプランナー視点であり、技術実装の詳細（クラス名・メソッド名等）は本文には含めず、補足情報として別途記載する

---

## 仕様書の構成

### UI仕様とゲームルール仕様の分離

仕様書は以下の2系統に分離されている。

| 系統 | 配置先 | 内容 |
|------|--------|------|
| UI仕様 | specs/ui/ | 画面構成・UI要素・フロー・遷移 |
| ゲームルール仕様 | specs/game-rules/ 等 | ゲームルール・システム仕様 |

- UI仕様書からゲームルール仕様書への参照は以下の形式で示される:
  > **ゲームルール仕様**: 仕様名（実際は該当仕様書へのリンクが貼られる）

### モックアップ

- モックアップはイメージ共有のための**参考資料**である
- モックアップ上の表示ラベル（ユーザー向け用語）と仕様テキスト上の名称（内部用語）は異なる場合がある
  - 表示用語は頻繁に調整が入るため、画像と仕様テキストの文言差異は許容事項とする
- 複数のモックアップ画像がある場合は `###` サブ見出しで区別する。サブ見出し名は描画対象の状態・ビュー・機能を示す名称とする（例: `### 商品一覧` / `### 購入確認ダイアログ`）
- モックアップ画像が1枚のみの場合はサブ見出しを付けず、`## モックアップ` 直下に画像を配置する

## 記法ルール

### パラメータ記法 `【】`

マスターデータ・コンフィグ等のデータ定義で管理すべきパラメータは `【】` でくくられている。

- 数値部分は `○` で示され、仕様書上では実数値に置き換えない
- `○` は「調整可能なパラメータが存在する」ことを意味する
- 具体値は継続的に調整が入る領域のため、仕様書の管轄外としている

```
例: 【カード予約列の上限○枚】
　→ 「カード予約列の上限枚数」というパラメータがデータ定義に必要であることを示す
```

`【】` の中身は口語で記述され、エンジニアが「どんなパラメータをコンフィグやマスターデータに追加すればよいか」判断できるようにしている。

### 暫定値 `（暫定: ○○）`

`【】` パラメータの後ろに `（暫定: ○○）` が付記される場合がある。

- 他の仕様記述に直接影響する（暫定値があったほうが説明しやすい）場合のみ付記される
- あくまで参考値であり、最終値ではない

```
例: 【リシャッフル時に最大HPの○分の○を失う】（暫定: 3分の1）
　→ 直後のテーブル（100→66→44→1）の算出根拠として暫定値が必要
```

### テーブルの親子関係（ツリー記号）

テーブル内で要素間に親子関係がある場合、ツリー記号で階層を表現する。

- `├` — 途中の子要素
- `└` — 最後の子要素

```
| 要素名 | 説明 |
|--------|------|
| ダイアログ | メニューダイアログ |
| ├ ボタンA | Aの説明 |
| └ ボタンB | Bの説明 |
```

### 操作・遷移一覧

画面仕様書の「操作・遷移一覧」は `| 操作 | 入力 | 備考 | 遷移先 |` の4列テーブルで記述する。

| 列 | 内容 |
|----|------|
| 操作 | ユーザーの意図・何が起こるかを端的に記述する |
| 入力 | [アウトゲーム入力マッピング（非公開資料）](private-notice.md)で定義された論理キー名をリンクとして記載する。入力がない場合（自動遷移等）は `-` |
| 備考 | 操作列だけでは不足する補足情報。不要な場合は `-` |
| 遷移先 | 画面遷移が発生する場合はリンクを記載。遷移なしの場合は `-` |

- 入力列には物理キー名（Xボタン、LB等）を直接使わず、[アウトゲーム入力マッピング（非公開資料）](private-notice.md)で定義された論理キー名を使用する。論理キー名はリンクとして記載し、クリックで入力マッピング定義にジャンプできるようにする
- インゲーム画面のプレイヤー操作はゲームルール仕様の範疇であり、[プレイヤーユニット — 操作一覧](game-rules/battle/spec-piece-player.md#操作一覧)で物理キーとともに定義される

### UI要素名の命名方針

UI要素一覧の「要素名」は以下の方針で命名されている。

- 要素名は「何の情報を表示するか（表示内容・役割）」を示す。表示手段（フェイス画像、アイコン等）は要素名に含めず、説明欄で記載する
- 要素名だけで「何の」情報か分かるよう、主語を省略しない

## 補足情報の種類

仕様本文とは別に、blockquote（引用ブロック）で補足情報が記載される場合がある。見出し行のフォーマットは `> **カテゴリ: 対象名**` で統一する。以下の4種類がある。

### 設計意図

仕様の設計判断の根拠を示す。[仕様策定原則](spec-principles.md)の該当節番号がある場合はリンクで参照する。該当する原則がない場合は「（該当原則なし）」と表記する。

> **設計意図: 対象名**<br/>
> - 理由の説明（[1.1 追い詰められながら上達していく体験](spec-principles.md#11-追い詰められながら上達していく体験)）
> - 理由の説明（該当原則なし）

### 技術補足

企画側から技術・設計に関して特別な言及を行う場合に使用する。

> **技術補足: 対象名**<br/>
> 補足内容

### 演出補足

UI要素一覧に収まらない演出仕様の補足。

> **演出補足: 対象名**<br/>
> - 補足内容

### 仕様補足

上記に該当しない仕様の追加説明。

> **仕様補足: 対象名**<br/>
> 補足内容

## 未決事項の範囲

仕様書末尾の「未決事項」セクションには、ゲームルール・機能仕様に関する未確定事項のみが記載される。以下は未決事項に含まない。

- 演出の詳細（トランジション、アニメーション速度、エフェクト内容等）
- UIレイアウト（表示位置、サイズ等）
- マスターデータ・テーブル設計（どの画面でどのマスターデータが必要か等）

これらはタスク管理の領域として別途管理される。<br/>
マスターデータの洗い出しはエンジニア責務とし、仕様書の `【】` パラメータを手がかりにタスク資料側で対応する。

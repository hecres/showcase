# コミット承認フロー

コミット実行前にユーザーの承認を得るための手順。`/commit` スキル経由・会話内での自然なコミットを問わず適用する。

---

## 承認依頼の手順

コミットを提案する際は、以下の4点を揃えた状態でユーザーに提示する。

### 1. type 選択（承認依頼前に必ず確認）

承認依頼でコミットメッセージを提示する **前** に、`.agents/skills/commit/references/git-commit-types.md` を参照し、変更内容に対する type が規約に沿っているか確認する。`/commit` スキル経由か自然対話か、承認スキップ指示があるかに関わらず、type 選択時には毎回参照する。

特に判断ミスを起こしやすい以下のケースに注意する。

- `#if UNITY_EDITOR || DEVELOPMENT_BUILD` 等で本番ビルドから除外されるデバッグ機能の修正・追加 → **`fix` / `feat` ではなく `chore`**
- `.Debug.cs` partial ファイルで隔離されたメンバーの修正・追加 → **`chore`**
- SR Debugger / エディタ拡張 / ツールスクリプト等の変更 → **`chore`**
- 動作に影響しないメンバー定義順の入れ替え・フォーマット変更 → **`refactor` ではなく `style`**

### 2. ステージング

コミット対象のファイルのみを `git add` でステージングする。未関係のファイルがステージに含まれていないことを確認する。

### 3. コミットメッセージ

予定しているコミットメッセージの全文を提示する。

### 4. 変更概要

変更内容の概要を簡潔に提示する。ステージング済みの差分から、何をなぜ変更したかが伝わる粒度で記述する。

## 提示テンプレート

以下のテンプレートに従い、区切り線と【】見出しで視認性を確保する。

```markdown
---

## 【コミットメッセージ】

\`\`\`
<type>: <description>

<body（任意）>
\`\`\`

---

## 【変更概要】

| 変更 | 内容 |
|------|------|
| ... | ... |

---

この内容でコミットしてよろしいですか？
```

## 分割コミット時の提示テンプレート

複数コミットに分割する場合は、各コミットを区切り線で明確に分離し、通し番号を付ける。

```markdown
---

## 【コミット 1/2】

\`\`\`
<type>: <description>
\`\`\`

| 変更 | 内容 |
|------|------|
| ... | ... |

---

## 【コミット 2/2】

\`\`\`
<type>: <description>
\`\`\`

| 変更 | 内容 |
|------|------|
| ... | ... |

---

この内容でコミットしてよろしいですか？
```

## コミット実行

ユーザーの承認を得てからコミットを実行する。承認前にコミットを実行してはならない。

## 承認スキップ

ユーザーがセッション内で「人間確認なしにコミットしてよい」等の承認スキップを指示した場合、そのセッション中は承認待ちを省略し、セルフレビュー完了後に即座にコミットを実行してよい。

コミット実行後、提示テンプレートと同じフォーマットで事後報告を行う。ただし末尾の確認文を以下に変更する。

```markdown
上記の内容でコミットしました。
```

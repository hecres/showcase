# ナレッジ改善

対話中に発生した「想定と異なる成果物」の原因を分析し、再発防止のための仕組み改善を行う。改善手段は rules, skills, agents, hooks, scripts 等から最適なものを選択する。

## 手順

- [ ] 1. **問題の特定** — 対話の中から以下を抽出する
  - 想定と異なった成果物は何か
  - 正しい成果物はどうあるべきだったか
  - どの対話で解決に至ったか
- [ ] 2. **根本原因の分析** — なぜ最初から正しい出力がされなかったかを掘り下げる
  - ルールが存在しなかった（知識の欠如）
  - ルールは存在したが参照されなかった（手順の欠如）
  - ルールは参照されたが従わなかった（強制力の欠如）
  - 上記以外の構造的要因
- [ ] 3. **改善手段の選定** — 根本原因に応じて最適な手段を選ぶ。強制力の強い手段ほど確実だが、過剰な制約は避ける
  - **ルール追加・修正**: 知識が不足していた場合。受動的にコンテキストに載る
  - **スキル作成・修正**: 手順が不足していた場合。明示的な呼び出しで手順を強制する
  - **エージェント作成・修正**: 自律的な検査・実行が必要な場合
  - **フック追加・修正**: 出力の構造的検証が可能で、回避不可能な強制が必要な場合
  - **スクリプト作成・修正**: 自動化・外部処理で解決できる場合
  - **上記の組み合わせ**: 単一手段では不十分な場合
  - **⛔ memory feedback は改善手段にしない。** memory はプロジェクトディレクトリとユーザーに紐づくローカル情報であり、別ディレクトリでのクローンや他ユーザーには共有されない。再発防止策は必ずリポジトリ内のルール・スキル等に反映する
- [ ] 4. **ユーザー承認** — 手順1〜3の結果を提示し、改善の実施前にユーザーの承認を得る
  - 特定した問題の一覧と、それぞれに対する改善手段を提示する
  - ユーザーは「その問題を改善すべきか」と「提案された改善手段が適切か」を判断する
  - 承認された問題・手段のみ実施に進む
- [ ] 5. **改善の実施** — 承認された手段を実装する
  - 新規ファイルを作成する場合は対応するテンプレートに従う
  - 既存ファイルを修正する場合は影響範囲を確認する
- [ ] 6. **セルフレビューを実施する**

## セルフレビュー

- [ ] 改善策は根本原因に対処しているか（表面的な対症療法になっていないか）
- [ ] 改善策の強制力は根本原因に見合っているか（過剰でも不足でもないか）
- [ ] 同じ問題が今後発生した場合、改善策によって最初から正しい出力が得られるか
- [ ] 改善策が既存のルール・スキル・フックと矛盾していないか
- [ ] 新規ファイルの場合、ファイル配置の原則（AGENTS.md）に従っているか

# コミットメッセージ記述規約

---

## description の記述ルール

- 変更内容を具体的に記述する
- git ログを後から読む第三者が、文脈なしで変更内容を理解できるようにする
- セッション内でのみ通用する一時的な用語を使用しない

### 禁止される表現

| 禁止 | 理由 | 代替 |
|------|------|------|
| 「A案」「B案」 | 対話中の選択肢番号 | 具体的な内容（例: 「操作モード統一」） |
| 「#3の対応」「評価項目#7」 | 議論番号 | 具体的な変更内容 |
| 「修正」のみ | 何を修正したか不明 | 「HP計算の丸め誤差を修正」 |
| 「更新」のみ | 何を更新したか不明 | 「UIの操作フロー更新」 |

### 良い例・悪い例

```
# 良い例 — 何が変わったか分かる
feat: 報酬の受領確認ダイアログを追加
fix: 画面遷移時にBGMが二重再生される問題を修正
refactor: Provider/Operator パターンに沿ってモジュールを分離
docs: メンバー定義順序規約にconst/static readonlyカテゴリ追加

# 悪い例 — 情報が足りない
feat: 機能追加
fix: バグ修正
refactor: リファクタ
docs: ドキュメント更新
```

## セルフレビューチェック項目

- [ ] `<type>: <description>` の形式になっているか
- [ ] type が type 一覧に含まれているか
- [ ] description が具体的で、第三者が文脈なしで理解できるか
- [ ] セッション固有の一時的な用語が含まれていないか
- [ ] 破壊的変更がある場合に `!` または `BREAKING CHANGE:` で明示されているか

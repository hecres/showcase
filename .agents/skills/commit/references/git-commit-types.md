# コミットメッセージ Type 規約

---

## type 一覧

### コードリポジトリ

| type | 用途 | 例 |
|------|------|-----|
| `feat` | 新機能の追加 | `feat: ユーザー設定画面を追加` |
| `fix` | バグ修正 | `fix: HP表示が0未満になる問題を修正` |
| `refactor` | リファクタリング（機能変更なし） | `refactor: 依存注入の整理` |
| `perf` | パフォーマンス改善 | `perf: 描画のバッチング最適化` |
| `test` | テストの追加・修正 | `test: 計算サービスのテスト追加` |
| `style` | コードスタイル変更（動作に影響しない） | `style: メンバー定義順をコーディング規約に合わせて整理` |
| `build` | ビルドシステム・外部依存の変更 | `build: UniTask を v2.5.0 に更新` |
| `ci` | CI 設定の変更 | `ci: GitHub Actions のビルドワークフロー追加` |
| `docs` | ドキュメントのみの変更（コード内コメント含む） | `docs: XMLコメント追加` |
| `chore` | 上記に該当しないメンテナンス。デバッグビルド専用機能の追加・変更を含む | `chore: SRDebuggerにプレイヤー回復デバッグ機能を追加` |

### ドキュメントリポジトリ

リポジトリ全体がドキュメント資産のため、type の使い分けが異なる。

| type | 用途 | 例 |
|------|------|-----|
| `docs` | 仕様書・設計書・タスク・skill/agent 定義・AGENTS.md・README 等の内容変更 | `docs: 画面仕様書追加` |
| `chore` | ドキュメント内容に関係しないメンテナンス | `chore: 補助スクリプト修正` |

- ドキュメントリポジトリではほぼ全てのコミットが `docs` に該当する
- `.agents/` / `.codex/` 内の skill / agent / rules 定義もドキュメント資産の一部として `docs` を使用する

## type の選択基準

判断に迷う場合は以下のフローに従う。

```
変更がある
  │
  ├─ 本番ビルドから除外されるデバッグ機能の変更？
  │    └─ YES → chore（`feat`/`fix` は使用しない）
  │
  ├─ 新しい機能が追加された？
  │    └─ YES → feat
  │
  ├─ 既存の不具合が修正された？
  │    └─ YES → fix
  │
  ├─ 外部から見た振る舞いは変わらない？
  │    ├─ パフォーマンスが改善された → perf
  │    ├─ テストのみの変更 → test
  │    ├─ コードの書式・スタイルのみ変更 → style
  │    └─ コード構造の改善 → refactor
  │
  ├─ ビルド・CI 設定の変更？
  │    ├─ ビルドシステム・依存関係 → build
  │    └─ CI/CD パイプライン → ci
  │
  ├─ ドキュメント・コメントのみ？
  │    └─ YES → docs
  │
  └─ 上記のいずれにも該当しない → chore
```

### feat と fix の境界

- 仕様通りに動いていなかったものを直す → `fix`
- 仕様になかった動作を新たに追加する → `feat`
- 既存機能の挙動を変更する → 意図による。バグ修正なら `fix`、仕様変更なら `feat`

### feat / fix は出荷される変更に限る

Conventional Commits の原則に従い、`feat` / `fix` は **本番ビルドに含まれエンドユーザーに出荷される変更** のみに使用する（SemVer の MINOR / PATCH を動かす性質の変更）。

以下のような **出荷対象外の変更** は、新機能・バグ修正であっても `chore` を使用する。

- `#if !HEC_UNITY_PRODUCT_BUILD` 等で本番ビルドから除外されるデバッグ機能
- SRDebugger の項目追加・変更
- エディタ拡張・ツールスクリプト等、実行時に含まれない資産
- `.Debug.cs` partial ファイルで隔離されたデバッグ用メンバーの追加・変更

```
chore: SRDebuggerにプレイヤーへのダメージ付与デバッグ機能を追加
chore: PieceObjectPresenterにReload/Rebuildの演出デバッグ機能を追加
```

### refactor と style の境界

- ロジックの構造変更（メソッド抽出、クラス分割、依存整理等） → `refactor`
- ロジックに触れない見た目の変更（フォーマット、空行、メンバー順序入れ替え等） → `style`

### docs の範囲

- コード内の XML コメントのみの変更 → `docs`
- コメント変更とコード変更が混在 → コード変更側の type を使用する

## 破壊的変更

公開 API や他モジュールとのインターフェースに破壊的変更がある場合、以下のいずれかで明示する。

### type に `!` を付加する方法

```
feat!: インターフェースを変更
```

### フッターに記述する方法

```
refactor: Provider を Provider/Operator に分割

BREAKING CHANGE: IHealthProvider を IHealthProvider（読み取り）と
IHealthOperator（書き込み）に分割。既存の参照箇所の修正が必要。
```
